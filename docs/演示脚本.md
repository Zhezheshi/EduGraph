# 演示脚本

本文档用于最终现场演示。演示目标是让评委在较短时间内看到完整闭环，而不是展示所有内部实现细节。

## 1. 演示主线

```text
教材管理 → 知识图谱 → 跨教材整合 → RAG 引用问答 → 教师反馈 → 整合报告
```

建议演示时间控制在 5-8 分钟。

## 2. 演示前准备

```text
1. 后端已启动。
2. 前端已启动。
3. DASHSCOPE_API_KEY 已配置。
4. 至少 7 本教材已注册。
5. 至少 2-3 本教材已解析。
6. 至少 2-3 本教材已构建知识图谱。
7. 跨教材整合已至少跑通过一次。
8. RAG 索引已建立。
9. report/整合报告.md 已生成或报告 Tab 可展示统计。
```

## 3. 页面展示顺序

### 3.1 打开工作台

展示内容：

```text
1. 左侧教材列表。
2. 中间知识图谱区域。
3. 右侧整合 / RAG / 对话 / 报告 Tab。
4. 顶部教材数、chunk 数、压缩比等状态。
```

讲解重点：

```text
系统不是单纯聊天机器人，而是从教材到知识图谱再到 RAG 问答的完整工作台。
```

### 3.2 教材解析

展示内容：

```text
1. 7 本教材已注册。
2. 展示其中一本教材的章节结构。
3. 强调 PDF 页码和章节元数据会进入 RAG 引用。
```

讲解重点：

```text
解析阶段保留教材名、章节名、页码，为后续引用和教学追溯服务。
```

### 3.3 单书知识图谱

优先展示：

```text
book_03 生理学
已构建前 2 章知识图谱：219 nodes, 187 edges
```

展示内容：

```text
1. 图谱节点数量和边数量。
2. 点击一个节点展示定义、章节、页码、来源。
3. 展示节点颜色或大小映射。
```

讲解重点：

```text
LLM 负责抽取概念和关系，系统把结果约束为结构化 JSON，前端可交互展示。
```

### 3.4 跨教材整合

展示内容：

```text
1. 选择 2-3 本教材运行整合。
2. 展示 merge / keep / remove 决策列表。
3. 展示压缩比、整合前后节点数。
4. 打开 1 个典型 merge 案例。
```

讲解重点：

```text
跨教材整合不是简单按名称去重，而是 embedding 召回 + LLM 复核，并保留整合理由。
```

### 3.5 RAG 引用问答

推荐问题：

```text
什么是炎症？
动作电位的产生机制是什么？
免疫应答和炎症反应有什么关系？
```

展示内容：

```text
1. 回答正文。
2. 引用来源列表。
3. 点击或展开原文 chunk。
```

讲解重点：

```text
回答只基于教材原文 chunk，每条引用包含教材、章节、页码，满足可追溯要求。
```

### 3.6 教师反馈

推荐输入：

```text
为什么把“炎症”和“炎症反应”合并？
我觉得“免疫应答”不应该删除，请保留。
把“抗原”和“免疫原”分开，它们不是同一个概念。
```

展示内容：

```text
1. 系统解释决策理由。
2. 系统根据反馈修改决策。
3. 决策列表或图谱发生变化。
```

讲解重点：

```text
教师反馈不是普通聊天，而是能修改 integration decisions 的人机协同闭环。
```

### 3.7 整合报告

展示内容：

```text
1. 原始教材数量。
2. 原始总字数。
3. 整合后字数。
4. 压缩比。
5. 合并 / 保留 / 删除数量。
6. 3-5 个重点案例。
7. 教学完整性说明。
```

讲解重点：

```text
报告数据来自系统实际运行结果，不是手工编造。
```

## 4. 备用演示路径

如果 LLM API 临时失败：

```text
1. 展示已缓存的 parsed JSON。
2. 展示已缓存的 graph JSON。
3. 展示已有整合决策。
4. RAG 可用关键词 / TF-IDF fallback。
```

如果图谱渲染失败：

```text
1. 展示节点和边 JSON。
2. 展示节点详情面板。
3. 展示整合决策和 RAG 引用。
```

如果部署失败：

```text
1. 本地前端 + 后端演示。
2. README 中说明启动命令。
3. 保留截图和测试记录。
```
