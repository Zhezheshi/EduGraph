# 需求分析

## 1. 任务理解

EduGraph Agent 面向多本医学教材的知识整合场景，目标不是单纯提供聊天问答，而是完成从教材解析、知识点抽取、知识图谱构建、跨教材去重压缩、RAG 引用问答到教师反馈修正的完整闭环。

系统需要解决的核心问题是：在多本教材内容重复、表述差异、结构不一致的情况下，自动抽取可教学复用的知识点，并在保留教学完整性的前提下压缩冗余内容。

## 2. 子问题分解

```text
教材文件 → 结构化章节 → 知识点与关系 → 单书知识图谱
→ 跨书语义对齐 → 整合决策 → 压缩统计
→ 基于原文 chunk 的 RAG 问答 → 教师反馈修正 → 整合报告
```

本项目将赛题拆分为以下子问题：

| 子问题 | 输入 | 输出 | 核心难点 |
|---|---|---|---|
| 多格式解析 | PDF / MD / TXT / DOCX | Textbook + Chapter | PDF 页码、目录、章节识别 |
| 知识点抽取 | 章节正文 | KnowledgeNode / KnowledgeEdge | LLM JSON 稳定性、知识点粒度 |
| 单书图谱 | 节点与边 | 可视化 graph JSON | 节点映射、关系类型规范 |
| 跨教材对齐 | 多本书节点 | 等价知识点组 | 同义概念、近义但不等价概念 |
| 内容压缩 | 对齐结果 | merge / keep / remove 决策 | 30% 压缩与教学完整性平衡 |
| RAG 问答 | 原文 chunks | 带引用回答 | 检索准确性、引用可追溯 |
| 教师反馈 | 自然语言指令 | 决策与图谱状态更新 | 意图解析、状态一致性 |
| 报告生成 | 系统统计与案例 | Markdown 报告 | 数据与系统运行结果一致 |

## 3. 知识点粒度定义

知识点不是任意实体名，而是适合教学组织的最小语义单元。一个知识点应满足：

```text
1. 可以被独立解释或教学讲解。
2. 在章节中具有明确概念、现象、机制、方法、公式或应用含义。
3. 可以与其他知识点建立 prerequisite / contains / parallel / applies_to 等关系。
4. 能够追溯到教材原文来源。
```

节点类型包括：

| 类型 | 说明 | 示例 |
|---|---|---|
| 核心概念 | 学科基础概念 | 炎症、动作电位、抗原 |
| 机制/过程 | 生理、病理或感染过程 | 炎症反应、免疫应答 |
| 现象 | 可观察的医学现象 | 发热、水肿、坏死 |
| 方法 | 检测、诊断或分析方法 | 细菌培养、革兰染色 |
| 应用 | 概念的临床或教学应用 | 抗体用于体液免疫 |

抽取时不追求“越多越好”，优先保证核心概念完整、定义清楚、关系可解释。

## 4. 重复判定标准

跨教材重复不是简单的名称相同，而是语义等价。系统将重复判定分为三层：

```text
第一层：规则匹配
- 名称完全相同
- 去除空格、标点、大小写后相同
- 中英文别名或医学常用别名匹配

第二层：向量相似度
- 使用 text-embedding-v3 生成 name + definition 向量
- 使用余弦相似度召回候选
- 默认候选阈值为 0.85

第三层：LLM 复核
- 对候选知识点进行 same / different 判断
- 输出 confidence 和 reason
- same 且 confidence >= 0.7 才自动合并
```

判定结果分为：

| 结果 | 处理方式 |
|---|---|
| 语义等价 | merge |
| 表述相关但不等价 | keep，并可建立 complement / applies_to 关系 |
| 上下位或前置依赖 | keep，并建立 contains / prerequisite 关系 |
| 低价值重复解释 | remove |

医学教材中许多概念来自不同学科视角，例如“炎症”在病理学、病理生理学、传染病学中的侧重点不同。系统不会只因为相关就合并，必须通过定义、上下文和 LLM 复核判断是否等价。

## 5. 教学连贯性保障

压缩不能破坏学习路径。系统采用以下保护规则：

```text
1. 被其他节点 prerequisite 指向的节点默认保留。
2. 多本教材共同出现的高频核心概念默认保留。
3. 删除节点前检查是否会使相关边孤立。
4. 对同一等价组优先保留定义更完整、出处更权威、上下文更系统的版本。
5. 教师反馈可以覆盖系统决策，支持 split / restore。
```

教学完整性主要通过图谱关系检查和报告中的案例解释来体现。整合报告需要说明典型合并案例为什么不会造成知识缺口。

## 6. 压缩比计算方式

赛题要求整合后体量不超过原始体量的 30%。本项目采用概念级压缩口径：

```text
原始体量 = 所有已解析教材正文 total_chars 之和
整合后体量 = 所有保留/合并知识点的 definition + explanation + source_summary 字符数之和
压缩比 = 整合后体量 / 原始体量
```

该口径的含义是：系统输出的是“面向教学复习与整合的精华知识库”，不是对原教材逐段改写。原文仍保留在 RAG 索引中，用于回答时引用和溯源。

## 7. RAG 分块策略

RAG 使用原文 chunk，而不是只使用整合后的知识点摘要。原因是赛题要求回答必须引用原文来源。

默认分块策略：

```text
chunk_size = 600
chunk_overlap = 80
top_k = 5
```

选择依据：

```text
1. 600 字通常可以覆盖一个局部概念的定义、解释和例子。
2. 80 字 overlap 可以降低知识点被切断的概率。
3. top-5 能在召回率和 prompt 长度之间取得平衡。
4. 每个 chunk 保留 textbook、chapter、page 元数据，保证引用可追溯。
```

检索方案：

```text
主方案：text-embedding-v3 + FAISS IndexFlatIP
回退方案：sklearn TF-IDF + cosine_similarity
```

生成约束：

```text
1. 只基于检索上下文回答。
2. 每个关键结论附带来源引用。
3. 找不到答案时回复“当前知识库中未找到相关信息”。
```

## 8. 教师反馈需求

教师反馈用于修正系统整合决策，不追求完全自由的开放式 Agent 操作。

支持的反馈类型：

```text
1. 解释：为什么合并 A 和 B？
2. 恢复：A 不应该删除，请保留。
3. 拆分：A 和 B 不是同一个概念，请分开。
4. 合并：A 和 B 应该视为同一知识点。
```

反馈处理流程：

```text
教师自然语言输入
→ LLM 解析为 action JSON
→ 后端修改 integration_decisions
→ 更新整合图谱和压缩统计
→ 保存对话历史
→ 前端刷新决策列表或图谱
```

## 9. 最小验收标准

P0 最小闭环必须满足：

```text
1. 至少 7 本教材可注册，至少 2-3 本教材可成功解析并展示章节。
2. 至少 2 本教材可生成知识点节点和关系边。
3. 图谱可视化支持点击节点查看详情。
4. 跨教材整合能生成 merge / keep / remove 决策。
5. 压缩比统计可显示，且口径在报告中说明。
6. RAG 能基于原文 chunk 返回带引用回答。
7. 教师反馈能修改至少一项整合决策。
8. README 和 docs 文档能指导本地运行。
```
